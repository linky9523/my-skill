# 意难平分支剧场

## 目标

把原作中的一个遗憾转化为可游玩的平行剧情。保留角色的核心动机和关系基础，让用户以故事导演身份不断选择，并让每次选择产生可以追踪的后果。

对外体验承诺：

> 从意难平的那一刻开始，亲手选择另一种结局。

## 强制原则

- 把用户对角色、关系和原作事实的说明视为最高优先级。
- 仅凭作品名建档时，先展示“原作基线草案”并等待用户确认；不得把不确定记忆写成事实。
- 用户提供原作文件时，只提取当前分支需要的人物、关系、事件和规则，不复述或拼接长段原文。
- 使用原创表达写新剧情。可以匹配题材氛围、叙事节奏和角色口吻，但不得承诺模仿在世作者的个人文风。
- 默认使用第三人称，单轮正文约 800–1500 个中文字；用户指定其他长度或视角时以用户要求为准。
- 每轮提供 3 个实质不同的选择。每个选择至少改变关系、信息、风险、目标或资源中的一项。
- 允许用户混合选项、自由输入第四条路线、暂停、重写、回溯或结束世界线。
- 不覆盖旧路线。每个节点保存自己的完整状态快照，回溯后只能继承目标节点及其祖先的状态。
- 先向用户展示状态更新摘要；获得确认后，才把新节点和状态写入项目。

## 启动判断

先识别用户处于哪种入口：

1. **仅报作品名**：执行“作品名建档”。
2. **提供原作文件**：执行“文件建档”。
3. **已有项目目录**：读取 `project.json`、`branch-tree.json`、`世界线状态.md` 和当前节点正文，再继续。
4. **只想试玩、不想建项目**：可以在对话中完成一轮，但在第二次选择前建议创建项目，以免世界线串线。

如果用户没有指定意难平节点，同时推荐 3–5 个候选节点。若用户已明确节点，不要强迫重新选择。

## 第一阶段：建立原作基线

### 作品名建档

根据已有知识整理一份简短草案，明确区分：

- `已知事实`：有较高把握且与当前分支直接相关。
- `待用户确认`：版本差异、冷门细节、时间顺序或记忆不确定项。
- `用户解释优先`：角色理解、CP 倾向、遗憾原因、禁忌内容。

一次只询问会阻塞开局的信息。向用户展示以下内容并等待确认：

```markdown
## 原作基线草案
- 作品与版本：
- 关键人物：
- 当前关系：
- 节点前已发生事件：
- 角色不可违背的核心动机：
- 待确认：
```

### 文件建档

读取用户提供的原作文件，先定位切入节点及其前置内容，再按需检索：

- 涉及角色的目标、恐惧、秘密和语言习惯。
- 节点前的人物关系、存亡与信息差。
- 会限制改写的世界规则。
- 与意难平直接相关的伏笔和未完成承诺。

不要为“完整”而把全文摘要塞进项目。对长篇文件记录来源位置，方便后续复核。

### 创建项目

用户确认原作基线和改写节点后，运行：

```bash
python3 <skill-dir>/scripts/project_tools.py init <project-dir> \
  --source-title "<作品名>" \
  --input-mode title \
  --branch-point "<切入节点>"
```

上传原作时将 `--input-mode` 改为 `upload`。随后填写 `原作基线.md` 和 `改写目标.md`；不要删除模板中的确认状态与边界字段。

项目协议和脚本命令详见 [references/project-schema.md](references/project-schema.md)。

## 第二阶段：确认改写契约

开局前与用户锁定四项：

1. **意难平**：用户真正想弥补的遗憾，而不只是事件名称。
2. **切入点**：从哪个决定或事件发生前开始。
3. **保留事实**：即使进入 IF 线也不能改变的设定。
4. **允许改变**：可以发生蝴蝶效应的事件、关系或结局。

可选补充：

- 偏好的角色或 CP。
- 希望避开的关系、情节和结局。
- 甜度、虐度、战斗强度和结局倾向。

如果用户让 Skill 推荐节点，候选节点必须说明：

- 改变的最小动作。
- 为什么符合角色动机。
- 最可能引发的连锁反应。
- 这条线可能付出的代价。

## 第三阶段：生成互动场景

生成前必须读取当前节点的状态快照。按照以下结构内部规划：

1. 当前角色各自想得到什么。
2. 本场景必须发生的状态变化。
3. 哪个原作惯性会阻碍理想结局。
4. 场景结尾把什么决定权交给用户。

正文应形成完整场景，而不是剧情提纲：

- 尽早建立地点、人物目标和即时压力。
- 让角色通过行动、选择和有目的的对白表现性格。
- 改变原作事件时保留阻力，不让理想结果无代价自动发生。
- 在结尾形成明确的新局面，再给选项。

场景后使用固定格式：

```markdown
## 本轮变化
- 关系：
- 信息：
- 风险：
- 未解决冲突：

## 请选择
**A. 选项标题**
行动：……
直接代价：……
可能打开：……

**B. 选项标题**
行动：……
直接代价：……
可能打开：……

**C. 选项标题**
行动：……
直接代价：……
可能打开：……

你也可以混合选项，或直接提出第四种行动。
```

设计选项前读取 [references/story-engine.md](references/story-engine.md)。禁止用“谨慎 / 激进 / 中立”三个近义姿态包装同一个结果。

## 第四阶段：确认并保存节点

用户选择后：

1. 先推演该选择造成的状态变化。
2. 向用户展示简短的“即将写入”摘要。
3. 用户确认后生成下一场景，并将正文保存到 `剧情/<节点ID>-<短标题>.md`。
4. 准备状态补丁和下一轮选项 JSON。
5. 使用脚本登记节点；不得手改节点 ID 或复制其他路线的状态。

```bash
python3 <skill-dir>/scripts/project_tools.py add-node <project-dir> \
  --parent "<父节点ID>" \
  --choice "用户选择摘要" \
  --title "场景标题" \
  --scene "剧情/<节点ID>-<短标题>.md" \
  --summary "本场景结果" \
  --state-patch '<JSON Merge Patch>' \
  --options '<JSON数组>'
```

脚本会生成稳定节点 ID、继承父节点快照、应用补丁、更新当前节点并重写 `世界线状态.md`。数组字段采用整体替换；不要把另一路线的完整状态作为补丁传入。

## 回溯、结局与导出

### 回溯

先列出分支树，再切换节点：

```bash
python3 <skill-dir>/scripts/project_tools.py list <project-dir>
python3 <skill-dir>/scripts/project_tools.py checkout <project-dir> --node "<节点ID>"
```

切换后重新读取 `世界线状态.md`。明确告诉用户已进入哪一个节点，不续接刚才路线中的独有事实。

### 结局

当主要冲突得到不可逆解决、核心人物弧光完成且当前选择的后果已兑现时，可以提出进入结局。不要因固定轮数强制结束。

结局节点使用 `add-node --ending` 登记。结局后提供：

- 生成后日谈。
- 回到关键节点尝试其他路线。
- 按当前世界线整理连续同人短篇。

### 路线整理

整理连续文本时，只拼接当前结局节点的祖先链。删除重复的互动提示和状态摘要，补充必要过渡，但不得混入兄弟分支事件。

## 质量检查

每轮提交给用户前检查：

- 角色的行动能否由既有欲望、恐惧或关系解释。
- 本轮是否至少改变了一项世界线状态。
- 三个选项是否会产生不同的短期代价和中期方向。
- 是否意外引用了其他分支的事实。
- 是否出现大段原作复述或近似复刻。
- 是否给用户保留自由输入和回溯权。

保存后运行：

```bash
python3 <skill-dir>/scripts/project_tools.py validate <project-dir>
```

如果验证失败，先修复项目结构或节点引用，再继续生成剧情。
