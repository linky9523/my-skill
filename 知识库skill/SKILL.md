# 知识库 Skill

## Skill 定位

这是一个帮助团队把项目资料沉淀为可读取、可检索、可复用知识库的 Skill。它主要解决三个问题：

1. 项目资料散落在不同文档里，后续很难找到。
2. AI 生成业务内容时缺少可靠上下文，容易胡编或泛化。
3. 新同事接手项目时，很难快速理解项目背景、产品信息和历史决策。

默认知识库位置为当前工作区下的 `knowledge_base/projects/{project_name}/`。如果用户已经指定目录，或仓库中已有资料目录规范，优先遵循用户指定位置和现有规范。

## 适用场景

用户出现以下需求时，使用此 Skill：

- 想为某个产品或项目建立资料库。
- 想把项目资料整理成统一目录。
- 想让 AI 后续基于项目资料回答问题。
- 想根据项目资料生成官网文案、PPT、卖点提炼、竞品分析等内容。
- 想快速了解某个项目的背景、定位、核心卖点和历史交付件。
- 想把旧项目资料沉淀为后续可复用资产。

## 不适用场景

- 用户只是临时问一个简单问题，不需要长期沉淀资料。
- 用户要创建新的 Skill，应转到 Skill 创作相关流程。
- 用户只是想学习如何使用 Skill，应转到 Skill 使用指引相关流程。
- 用户没有提供任何项目资料时，不要编造项目事实。

## 知识库目录结构

建议结构：

```text
knowledge_base/
└── projects/
    └── {project_name}/
        ├── project_index.md
        ├── 01_product_info/
        ├── 02_selling_points/
        ├── 03_competitors/
        ├── 04_user_research/
        ├── 05_marketing_outputs/
        ├── 06_meeting_notes/
        └── 99_archive/
```

目录用途：

- `01_product_info`：产品基础信息、定位、参数、版本、价格、上市节奏等。
- `02_selling_points`：卖点信息表、功能说明、用户收益、技术解释、传播表达等。
- `03_competitors`：竞品资料、竞品卖点、价格配置对比、传播打法分析等。
- `04_user_research`：用户调研、访谈纪要、评论分析、目标人群洞察等。
- `05_marketing_outputs`：官网框架、PPT、发布会文案、门店话术、社媒内容、培训材料等。
- `06_meeting_notes`：会议纪要、评审意见、关键决策、修改记录等。
- `99_archive`：过期资料、旧版本文件、已废弃方案等。

## 工作流程

### 第一步：确认项目名称

如果用户没有提供项目名称，先提醒用户提供一个项目名，例如：

- Find X10e
- Reno16 Pro
- 官网框架项目
- 产品策划白皮书
- 某次发布会项目

不要在项目名缺失时自行编造目录名。可以临时建议使用用户文件名或业务名，但必须让用户确认。

### 第二步：建立项目文件夹结构

为该项目创建统一的资料目录，并说明每个文件夹应该放什么内容。创建前先检查当前仓库是否已有 `knowledge_base/`、`projects/`、`docs/`、`materials/`、`references/` 等资料目录规范；如已有规范，沿用现有规范，只补缺少的项目层级和索引文件。

### 第三步：生成项目索引文件

在项目文件夹下创建 `project_index.md`，用于记录：

- 项目名称
- 项目背景
- 产品定位
- 目标用户
- 核心卖点
- 关键竞品
- 主要交付件
- 重要时间节点
- 资料目录说明
- 最后更新时间

如果项目资料不足，索引中使用 `待补充`，不得虚构事实。

### 第四步：读取资料时优先遵循知识库

当用户基于项目提问时，必须优先读取对应项目文件夹中的资料。读取顺序建议：

1. 先读 `project_index.md`，确认项目背景、资料范围和更新时间。
2. 按问题类型读取对应资料目录，例如卖点问题优先读 `02_selling_points/` 和 `01_product_info/`。
3. 若问题涉及历史版本或废弃方案，再读取 `99_archive/`。
4. 如资料之间冲突，列出冲突点和来源，不直接替用户选择“最新事实”。

回答时要区分：

- 知识库中明确有依据的信息。
- 根据资料推理出来的判断。
- 知识库中没有、需要用户补充的信息。

### 第五步：输出业务内容时必须引用项目资料

如果用户要求生成官网文案、PPT、卖点、竞品分析、传播表达、产品介绍、培训话术等内容，优先基于知识库资料，而不是凭空生成。输出中要说明使用了哪些资料；关键产品事实、参数、时间、竞品对比和能力宣称必须能回到知识库依据。

### 第六步：支持知识库更新

当用户新增资料时，帮助用户：

- 判断资料应该放在哪个目录。
- 更新 `project_index.md`。
- 总结新增资料的核心信息。
- 标记过期资料或归档资料。

新增资料时优先保留原文件，不覆盖旧文件。若资料明显过期，移动或建议移动到 `99_archive/`，并在索引中说明旧版本状态。

## 输出格式

### 当用户要求建立项目知识库时

输出：

1. 项目知识库路径。
2. 建议创建的文件夹。
3. 每个文件夹应该放什么。
4. 初版 `project_index.md`。
5. 后续使用方式。
6. 注意事项。

### 当用户基于知识库提问时

输出：

1. 结论。
2. 依据的资料。
3. 推理过程。
4. 可用于业务的输出。
5. 知识库缺失的信息。

如果用户只是要一个简短答案，可以压缩结构，但仍要保留“依据”和“缺失信息”的判断。

## 重要限制

- 不允许编造知识库中没有的产品事实。
- 不允许把旧版本资料当成最新资料，除非用户明确要求看历史版本。
- 如果资料之间冲突，要提醒用户存在冲突，并列出冲突点。
- 如果没有读取到相关资料，要明确说明“当前知识库未找到依据”。
- 不要默认联网搜索，除非用户明确要求。
- 对保密资料只在本地仓库或用户指定环境中处理，不要主动上传到外部平台。
- 引用用户内部资料时，只概述必要信息；不要把整份敏感原文复制到回答里。

## 模板文件

- `knowledge_base_template.md`：用于向团队说明项目资料怎么放、怎么命名、如何做版本管理。
- `project_index_template.md`：用于创建每个项目的 `project_index.md`。

## 示例

### 示例 1：为 Find X10e 建立项目知识库

用户说：“帮我给 Find X10e 建一个项目知识库。”

处理方式：

1. 在当前工作区检查是否已有知识库规范。
2. 创建 `knowledge_base/projects/Find_X10e/`。
3. 创建 `01_product_info` 到 `99_archive` 的目录。
4. 复制 `project_index_template.md` 的结构生成 `project_index.md`。
5. 在索引中将项目名称写为 `Find X10e`，未知内容标记为 `待补充`。
6. 告诉用户后续可把参数表放入 `01_product_info/`，卖点表放入 `02_selling_points/`，会议纪要放入 `06_meeting_notes/`。

### 示例 2：基于项目知识库生成官网卖点框架

用户说：“基于 Find X10e 知识库，帮我生成官网卖点框架。”

处理方式：

1. 先读取 `knowledge_base/projects/Find_X10e/project_index.md`。
2. 再读取 `01_product_info/`、`02_selling_points/`、`05_marketing_outputs/` 中相关资料。
3. 输出官网框架时标明依据资料，例如“依据：卖点信息表、产品参数表、上一版官网框架”。
4. 对没有依据的卖点、参数或传播词标记为 `待确认`。
5. 不主动补充网上资料，除非用户明确要求联网搜索。

### 示例 3：新增一份竞品分析文件后，更新项目索引

用户说：“我新增了一份 X 品牌竞品分析，帮我放进知识库并更新索引。”

处理方式：

1. 判断该文件应放入 `03_competitors/`。
2. 保留原文件名；如文件名混乱，建议命名为 `YYYY-MM-DD_竞品名称_竞品分析_版本.md/xlsx/docx`。
3. 阅读新增资料，提炼关键竞品、对比维度、价格配置、传播打法和待确认项。
4. 更新 `project_index.md` 中的“竞品”“关键资料列表”“关键决策记录”或“待补充资料”。
5. 如果新增资料与原有资料冲突，在索引中新增“资料冲突记录”。
